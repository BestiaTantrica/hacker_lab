ANÁLISIS DE discovery_pasivo.py
--------------------------------------------------------------------------------

Código (1648 bytes, Python 3.12.3):
  - Lee dominios de config/objetivos.txt
  - Consulta crt.sh con URL: https://crt.sh/?q=%.DOMINIO&output=json
  - Timeout: 15 segundos
  - Deduplica subdominios con set()
  - Elimina comodines (*.) de los resultados
  - Guarda JSON en resultados/discovery_crudo.json

LO QUE HACE BIEN:
  - Diseño modular y legible
  - Timeout rígido para evitar cuelgues
  - Deduplicación de resultados
  - Rutas absolutas mediante os.path.expanduser

GAPS IDENTIFICADOS:
  GAP 1: Fallo silencioso
    El bloque except captura errores y retorna [].
    El JSON resultante tiene arrays vacíos sin distinguir
    "no hay subdominios" de "error de red o timeout".

  GAP 2: Sin logging persistente
    Todo va a stdout. Si se ejecuta vía cron, el output se pierde
    a menos que se redirija (>> logs/discovery.log).

  GAP 3: Nombre de archivo fijo
    OUTPUT_FILE = "discovery_crudo.json" (siempre el mismo).
    Cada ejecución sobreescribe la anterior sin historial.
    El CONTRATO_SUPERVISOR menciona "actual.json" (inconsistencia).

  GAP 4: Sin validación de body vacío
    No verifica si response.text está vacío antes de parsear JSON.
    Un body vacío genera una excepción no manejada del lado del caller.

RESULTADO DEL ÚLTIMO RUN (2026-06-29T23:37:55 UTC):
  {
    "timestamp": "2026-06-29T23:37:55.476299",
    "dominios": {
      "starbucks.com": []
    }
  }
  DIAGNÓSTICO: starbucks.com devolvió array vacío.
  starbucks.com tiene miles de entradas en crt.sh — resultado anómalo.


--------------------------------------------------------------------------------
DIAGNÓSTICO: POR QUÉ starbucks.com DEVUELVE []
--------------------------------------------------------------------------------

PRUEBA 1 realizada:
  curl -s "https://crt.sh/?q=%.starbucks.com&output=json" | python3 -c "..."
  RESULTADO: JSONDecodeError (body vacío)

PRUEBA 2 realizada:
  curl -v "https://crt.sh/?q=%.starbucks.com&output=json" 2>&1 | head -40
  RESULTADO: TLS handshake COMPLETADO. Conexión establecida.
    - Host resuelto: 91.199.212.73 (IPv4)
    - TLS 1.2 con ECDHE-RSA-CHACHA20-POLY1305
    - Certificado verificado OK
    - El output fue cortado por head -40 antes del HTTP response code

CONCLUSIÓN:
  La red de OCI SÍ tiene acceso a crt.sh (TLS funciona).
  El body vacío NO es un bloqueo de IP.
  HIPÓTESIS MÁS PROBABLE: timeout de query en el servidor de crt.sh.
  crt.sh tiene límites propios de tiempo de ejecución de base de datos.
  Las queries sobre dominios grandes (starbucks.com = miles de entradas)
  pueden exceder ese límite y devolver body vacío con HTTP 200.

PRUEBA PENDIENTE (para confirmar):
  curl -s -o /dev/null -w "HTTP: %{http_code} | Size: %{size_download} bytes | Time: %{time_total}s\n" \
    "https://crt.sh/?q=%.starbucks.com&output=json"

  Si muestra: HTTP 200 | 0 bytes | ~15s -> confirma timeout de crt.sh
  SOLUCIÓN: Usar dominios más pequeños en objetivos.txt, o agregar
  reintentos con backoff al script. La arquitectura NO necesita cambio.
